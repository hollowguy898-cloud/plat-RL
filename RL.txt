Here's a clean, practical plan to build your XLand-style open-ended RL playground using Jax2D as the physics backbone.
We will keep two separate games (not one combined world):
* Celeste Mode — Precision platforming focus (tight movement, dashes, variable jump, wall jumps, one-way platforms, spikes, moving platforms, crystals/berries as collectibles).
* Hollow Knight Mode — Combat + exploration focus (nail attacks, pogo bouncing on enemies, spells, geo collection, simple enemy AI, bosses, acid pools, etc.).
At setup/env creation, you choose the game mode via a config flag (e.g. game="celeste" or game="hollow_knight"). This keeps the code modular, easier to tune physics & rewards per game, and still allows shared infrastructure for procedural generation, training loops, and meta-RL.
Overall Architecture (Pure JAX style)
* EnvState: A single flax.struct.dataclass (or pytree) that holds everything — player, physics bodies, level geometry, timers, game-specific flags, RNG, etc.
* reset(key, config) → returns initial state + obs
* step(state, action, key) → new_state, reward, done, info
* Everything jit + vmap compatible for thousands of parallel envs.
* Use Jax2D for all rigid-body simulation (player as dynamic body, platforms as static or kinematic, enemies as dynamic bodies, etc.).
* Add game-specific logic on top of the physics step using jax.lax.cond, select, while_loop/scan for things like coyote time, input buffering, dash state machines.

Hollow Knight Mode World (Combat + Metroidvania lite)

   * Nail slash (short range attack, cooldown, can be directional or omnidirectional).

   * Bosses: larger bodies with patterns (one or two attack phases).

Key Principles
* Everything is a pytree → fully jit/vmap compatible.
* Jax2D handles rigid-body collisions, integration, impulses.
* Custom game logic (coyote time, dash, one-way platforms, variable jump) lives in pure JAX functions with lax.cond/select.
* Dynamic scenes: different number of platforms/enemies per env is ok, but for clean vmap we pad + use masks.
* No rendering, no graphics.
World for Celeste Mode (as close as possible, simplified for RL)
* Units: World ~ 40m wide × 30m tall (Jax2D uses meters-like units; tune gravity to ~ -30 to -40).
* Player: Capsule shape (radius ~0.4, height ~1.2) or box with rounded feel. Dynamic body.
* Terrain: Static polygons/boxes for solid ground/walls. Kinematic bodies for moving platforms.
* One-way platforms: Handled in post-physics (inspired by classic Box2D patterns): after scene.step(), check player downward velocity and contact normal; disable response if player is coming from below or pressing down.
* Hazards: Static spike polygons → overlap = death.
* Collectibles: Small static circles ("strawberries") → overlap = +reward + remove (mark as collected via mask).
* Goal: Static sensor rectangle → overlap = big reward + terminal.
* Procedural Levels (XLand flavor):
   * Start on left platform, goal on right/top.
   * Generate sequence of gaps, floating platforms, vertical climbs, spike pits.
   * Vary difficulty: gap width, platform spacing, moving platforms, dash-required sections.
   * Task sampler: "reach goal", "collect K berries", "reach goal under time limit", with curriculum based on success rate.
Player Mechanics (the "nail the feel" part)
* Ground vs air acceleration/friction.
* Variable jump (strong initial impulse, reduce while holding).
* Coyote time (grace period after leaving ground).
* Jump buffer (queue jump for ~6–8 frames).
* Dash: 8 directions (discrete or continuous), 1 use in air, refills on ground. Instant velocity boost + short cooldown.
* Wall slide (slow fall) + wall jump (angled kick).
These will be state variables in EnvState and updated with lax.scan or simple cond chains.
Observations (~40–60 dims, compact vector)
* Player: pos (2), vel (2), on_ground (bool), coyote_remaining, dash_remaining, facing, timers (dash_cooldown, jump_buffer).
* Raycasts: 8–12 rays (down for ground, forward, diagonals) → distances to terrain/hazard.
* Nearest items: relative pos to goal + 2–3 collectibles (with mask).
* Task: distance to goal, time left, collected count.
Rewards
* Sparse: +1000 reach goal, -200 death, +50 berry.
* Shaped: small progress toward goal or berry, alive bonus optional.

* Health scaling: Base HP as above. Multiply by difficulty factor (1.0 early → 2.5+ late). Use integer HP.
* Damage to player: Keep 4–15 range. Player has masks (health), so balance for 1–3 hits per mistake.
* Speed: Define in world units (e.g. player max run = 10.0). Slow=3–5, Medium=6–9, Fast=10–14, Very Fast=15+.
* AI State Machine (simple FSM in JAX with lax.cond or scan):
   * Patrol (back-forth or random)
   * Aggro (move toward player when in range)
   * Attack (cooldown-based)
   * Recover / idle
* Pogo detection: When player attacks downward + overlaps enemy top → apply upward impulse to player + damage enemy.
* Procedural spawning: For a level, sample 3–8 enemies from a pool based on "biome" (Crossroads pool, Greenpath pool, Deepnest pool, etc.). Add masks for different numbers in vmap.

### 1. Player (The Knight)
**Hurtbox** (what enemies hit):
- Shape: **Capsule** or rounded rectangle (best for platformers in Jax2D).

  - Height: 1.4 – 1.6 (tall but narrow — the "small hurtbox" feeling)
  - Offset: Slightly above feet (head is less vulnerable in practice).

**Nail Attack Hitbox** (what the player hits with):
- Shape: **Rectangle** or thin box extending forward.
- Size: Generously large — width 2.0–2.5 (horizontal), height 1.2–1.5.
- Direction: Forward + slight upward/downward variants for pogo.
- Downward (pogo) hitbox: Smaller rectangle or circle below the player (key for enemy bounces).
- Notes: The nail swing hitbox is famously bigger than it looks — this is a big part of the satisfying combat feel.


**Other**:
- Spell hitboxes (e.g. Vengeful Spirit): Projectile rectangles or circles.
- Dash: No damage, but temporary invuln or reduced hurtbox.

How NPC Talking Actually Works in Celeste
* Most "NPCs" (Theo, Granny, Oshiro, Badeline appearances, phone calls, etc.) are triggered by:
   * Walking into a trigger zone (invisible rectangle).
   * Pressing the up direction (or sometimes just proximity) near certain objects (e.g. Theo at campfires, payphones).
* Dialogue is non-interactive for the player in the base game — it's a cutscene-style sequence that plays out automatically once triggered. You just stand there and watch/read.
* After the dialogue finishes, the NPC usually disappears, gives a small hint, or the level continues. No branching choices in the vanilla game (except minor flavor).
* Important cases where it matters for movement:
   * Theo in some levels gives encouragement or minor story.
   * Payphone in Old Site triggers a major story beat + Badeline chase.
   * Some objects (like memorial, cassettes, strawberries in certain spots) have short flavor text when approached.
In short: talking is passive — you walk up and it happens. The risk for RL agents is getting stuck in a loop trying to "solve" the NPC instead of continuing platforming.
How We Should Handle NPCs / Talking in Our Jax2D Env (Headless RL)
We want to keep it simple, fast, and anti-stuck while still feeling faithful to Celeste.

      * dialogue_id or short flag (e.g. "talked" bool per NPC).
   * When player overlaps the trigger zone → automatically "talk" (no extra action needed).
2. Interaction Mechanic (Anti-Stuck)
   * Option A (Recommended for simplicity): Auto-trigger on overlap + face the NPC for 1–2 seconds.
      * Agent gets a small shaped reward (+10–30) for "talking".
